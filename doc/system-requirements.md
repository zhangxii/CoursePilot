# CoursePilot 系统需求规格说明书

## 1. 文档目的

本文将 `doc/task.md` 转化为可验收的系统需求，作为架构设计、模块设计、开发和测试的共同基线。目标产品不是单向生成答案的聊天窗口，而是围绕作业正式版本持续协作的工作区：用户可以上传自己的稿件或让 Agent 生成候选稿，在彼此隔离的对话中探索不同方案，经独立自动审查和定向优化后，由用户决定是否采用为新的正式版本。

## 2. 产品范围

### 2.1 范围内

- 使用自然语言完成课程笔记总结、作业完成、作业评审和作业修改。
- 由 1 个课程学习主 Agent 统一编排 4 个专业 Agent。
- 默认只检索当前课程资料，满足条件时才检索历史课程资料。
- 上传预先整理好的 Markdown 或 UTF-8 纯文本课程资料，并保留其中的章节和来源信息。
- 上传用户已有的 `.md`、`.txt` 或 `.docx` 作业稿件，并纳入版本管理；PDF 作业稿暂不在首版范围内。
- 创建、切换、重命名、分支和归档同一道题下的多条对话，并显式选择对话所依据的作业版本。
- 输入或上传优化方向；未提供方向时由 Agent 先分析问题并提出可选择的优化建议。
- 生成和优化任务默认经过独立自动审查，生成结果先作为候选稿，由用户确认后才进入正式版本链。
- 使用 JSONL 保存各对话消息，使用 Markdown/YAML 持久化业务记录。
- 通过流程导航、空状态说明、动作型按钮和下一步提示引导用户完成工作。
- 对 Agent、模型和工具调用进行链路追踪。

### 2.2 范围外

- 复杂角色权限、组织级租户隔离和生产级计费；首版只支持一个项目小组及其成员。
- LangGraph、跨进程长流程恢复和多人实时协同编辑。用户采用候选稿属于产品内确认动作，不属于外部审批工作流。
- 自动接入教务系统、在线提交作业或替代教师给出正式成绩。
- OCR、音视频转写和复杂公式/图表语义识别。

## 3. 用户与核心场景

主要用户为共同完成课程大作业的学生小组。系统围绕一个小组、多道作业题、每道题的一条正式版本链以及多条独立对话工作；课程资料为当前选中的题目提供知识依据。候选稿可以并存，但任一时刻每道题只有一个当前正式版本。

| 场景         | 用户目标                   | 预期结果                               |
| ------------ | -------------------------- | -------------------------------------- |
| 课程资料导入 | 建立可检索课程知识库       | 文件正文、课程元数据、页码和来源保存在本地 |
| 小组初始化   | 建立协作上下文             | 记录小组名称和成员                     |
| 题目管理     | 创建并切换小组作业题       | 当前题目、要求和评分标准               |
| 笔记总结     | 提炼当前课程内容并用于实践 | 结构化笔记、课程依据、必要的前序知识   |
| 作业完成     | 结合课程进展生成作业候选稿 | 任务理解、候选稿、课程依据、不确定项   |
| 用户稿上传   | 将线下修改或已有作业带回系统 | 可追溯来源的新正式版本或参考附件       |
| 作业评审     | 独立评价正式版本或候选稿   | 分项评分、扣分依据、追问和修改优先级   |
| 作业修改     | 按方向优化指定基础版本     | 候选稿、逐项修改说明、问题闭环状态     |
| 方案分支     | 分开探索 Agent 方案和用户方案 | 多条隔离对话，各自绑定明确的基础版本 |
| 定向优化     | 按用户方向或 Agent 分析改稿 | 经自动审查的候选稿与版本差异           |
| 新手引导     | 看懂当前状态和下一步       | 流程进度、解释性文案和明确操作按钮     |

## 4. 功能需求

### 4.1 课程与资料管理

- **FR-001 课程管理**：系统应支持创建、查看和切换课程，并且任一会话至多有一个当前课程。
- **FR-002 元数据管理**：每份资料应至少包含 `course_id`、`course_name`、`course_date`、`teacher`、`topic`、`material_type`、`status`；`status` 取值至少支持 `current` 和 `archived`。
- **FR-003 文件导入**：系统应支持 UTF-8 `.md` 和 `.txt`；课程资料由用户预先解析，系统不处理 PDF/PPTX。
- **FR-004 页码保留**：预处理结果应转换为带页码标题的 Markdown，使检索结果能够定位原文件及页码。
- **FR-005 本地资料持久化**：系统应为上传内容自动添加 YAML Front Matter，并统一保存为独立 Markdown 文件。当前课程资料总量未超过字符预算时直接提供正文；超过预算时按章节执行本地关键词检索，Front Matter 不发送给模型。
- **FR-006 当前课程标记**：课程切换后，检索过滤条件和会话上下文应同步更新，不得继续默认使用旧课程。

### 4.2 小组、题目与正式答案

- **FR-007 小组管理**：系统应维护一个项目小组的名称和成员名单；首版不支持多个小组并行管理。
- **FR-008 多题目管理**：系统应支持为同一小组创建、查看、更新和切换多道作业题；任一会话至多有一道当前题目。
- **FR-009 每题单一正式版本**：每道题至多有一份当前正式答案。用户采用候选稿或上传新稿时产生新版本并替换当前正式版本；未采用的候选稿可以并存，但不得被读取为正式答案。

### 4.3 受控检索

- **FR-010 当前课程检索**：提供 `search_current_course(query)`，强制以会话中的 `active_course_id` 过滤资料。
- **FR-011 历史课程检索**：提供 `search_course_archive(query)`，排除当前课程并限定归档资料。
- **FR-012 当前优先策略**：所有学习任务默认先检索当前课程，不允许专业 Agent 直接无过滤检索全部资料。
- **FR-013 历史检索条件**：只有用户明确要求、当前资料引用前序知识、当前资料不足或修改需要检查跨课程一致性时，才允许历史检索。
- **FR-014 当前标准优先**：历史资料不得覆盖当前课程教师给出的定义、标准和任务要求。
- **FR-015 检索证据**：检索结果应携带文件名、课程、页码或章节、片段等可引用信息。

### 4.4 Agent 编排

- **FR-020 意图识别**：主 Agent 应识别总结、完成、评审、修改等意图；缺少关键输入时应指出缺失项。
- **FR-021 课程确定**：主 Agent 应从显式输入或会话上下文确定当前课程；无法可靠确定时不得猜测。
- **FR-022 专业 Agent 调用**：主 Agent 应将 4 个专业 Agent 作为工具调用，并保留最终控制权和响应整合职责。
- **FR-023 多 Agent 组合**：主 Agent可按任务需要顺序调用多个专业 Agent。生成和优化流程必须包含独立自动审查；审查发现可自动修复的问题时，应进入一次有界修正并再次核对，不得无限循环。
- **FR-024 笔记总结**：笔记 Agent 应生成课程问题、核心概念、分析方法、案例、常见错误、教师标准、实践用法和前序关系，并标注来源。
- **FR-025 作业完成**：作业 Agent 应读取当前题目及对话绑定的基础版本，先分析任务和课程标准，再生成候选稿；输出任务理解、候选答案、课程依据和不确定信息，不得未经用户确认直接替换当前正式版本。
- **FR-026 独立评审**：评审 Agent 仅接收题目、当前课程资料、教师要求、评分标准和待评答案，不接收生成 Agent 的中间推理。
- **FR-027 评审可解释性**：每项扣分应给出分值、位置、依据、原因和修改建议。
- **FR-028 作业修改前置条件**：修改 Agent 必须读取优化任务指定的基础版本、用户优化方向和对应审查结果；不存在审查时应先自动触发独立审查。
- **FR-029 修改模式**：修改应支持“保留原方案并优化”和“允许重组方案”。前者保留核心观点与主要结构，后者允许调整结构和替换薄弱方案；两者都必须遵守用户声明的保留项、禁止项、格式和篇幅限制。

### 4.5 上下文与业务记录

- **FR-030 对话上下文**：每条对话应维护 `conversation_id`、`active_course_id`、`team_id`、`active_assignment_id`、`base_answer_version_id` 和可选 `parent_conversation_id`；题目、正式版本、评审等业务事实始终从业务记录读取。当前课程、当前题目、当前对话是三个独立选择轴。
- **FR-031 对话持久化**：每条对话使用独立 JSONL Session，重启后能够恢复；系统应支持新建空白对话、基于正式版本新建对话、从现有对话创建分支、切换、重命名和归档。新对话默认继承当前正式版本，但不继承其他对话消息；分支必须记录父对话和精确的消息分支点，并复制截至该点的消息快照。
- **FR-032 业务持久化**：课程资料、作业和答案正文保存为 Markdown；工作区、课程、评审和修改保存为 YAML；不使用数据库，也不以聊天历史代替业务记录。
- **FR-033 正式答案版本**：每道题可拥有多个正式答案版本；采用候选稿或上传用户新稿时创建新版本，保留基础版本、来源、审查、优化模式、操作成员和说明的关联。历史正式版本不可被覆盖。
- **FR-034 状态一致性**：当前题目、当前正式版本、候选稿状态、对话绑定版本和最近评审更新应保持一致；失败时不得留下半更新状态，也不得串用其他题目或对话的记录。

### 4.6 作业稿件、对话与优化

- **FR-044 用户作业上传**：系统应支持上传非空 `.md`、UTF-8 `.txt` 和 `.docx` 作业稿件，校验类型、大小和安全文件名，保存原件及规范化正文。
- **FR-045 上传用途**：上传时应明确选择“设为初版”“作为新正式版本”“仅作参考附件”；已有正式版本时不得静默覆盖。“设为初版”和“作为新正式版本”均须记录版本说明与来源 `user_upload`。
- **FR-046 候选稿**：Agent 生成或优化的结果应先保存为候选稿，状态至少包含 `draft`、`ready_for_adoption`、`adopted`、`discarded`、`superseded`；候选稿不是正式答案，不改变其他对话的基础版本。“继续修改候选稿”应创建派生候选稿并保留来源候选稿，来源状态改为 `superseded` 且记录替代者。
- **FR-047 采用候选稿**：用户点击“采用为正式新版本”后，系统应以事务创建下一正式版本并关联候选稿、基础版本、审查和操作成员；采用前应明确提示将产生的版本号且不会覆盖历史版本。
- **FR-048 版本比较**：系统应展示基础版本与候选稿或两个正式版本之间的正文差异、修改摘要、已解决问题和未解决问题。
- **FR-049 对话隔离**：同一道题可以有多条对话；每条对话只读取自身消息和显式绑定的基础版本，不得因切换对话而隐式引用其他对话内容。
- **FR-050 优化方向输入**：优化任务应支持直接输入文字或上传 `.md`、UTF-8 `.txt`、`.docx` 方向文件，并保存规范化内容、原件和来源。
- **FR-051 Agent 问题分析**：用户未提供优化方向时，Agent 应先分析基础版本，输出问题、理由、影响和优先级，允许用户选择全部、部分或补充方向后再优化。
- **FR-052 优化任务**：每次优化必须显式绑定一个基础正式版本或基础候选稿（二者恰好一个），并记录用户方向、Agent 建议、选中方向、修改模式、保留项、禁止项、格式和篇幅限制。
- **FR-053 自动审查**：生成和优化候选稿必须由与生成上下文隔离的 ReviewAgent 自动审查；审查摘要应展示发现数、已修复数和仍需确认数。自动审查不创建正式答案版本。
- **FR-054 正式评审**：用户可对任一正式版本发起正式评审；正式评审与自动审查使用同一评审契约，但目的、触发方式和持久化类型必须可区分。

### 4.7 界面与可观测性

- **FR-040 Streamlit 界面**：应提供小组与成员信息、题目和课程切换、课程资料、作业稿件与版本、对话列表、优化任务、审查结果和执行状态反馈。
- **FR-041 结构化输出**：关键 Agent 输出使用 Pydantic 模型校验；校验失败应被捕获并向用户提供可恢复反馈。
- **FR-042 Tracing**：应记录模型生成、专业 Agent 调用、函数工具调用、过滤条件、异常和耗时。
- **FR-043 错误处理**：文件校验、本地保存、检索和模型调用失败时，应展示可理解错误并记录诊断信息。
- **FR-055 首次使用引导**：首次进入时应让用户选择“已有作业稿”“只有题目和资料”“先整理课程资料”，并进入对应流程。
- **FR-056 流程与空状态**：首页应展示“创建作业→准备资料/稿件→获得初版→自动审查→定向优化→确认版本”的进度；每个空状态必须解释用途并提供可执行的下一步。
- **FR-057 动作型措辞**：主要按钮应采用“动作 + 结果”的措辞，并在创建正式版本等关键操作前说明影响；不得用未连接业务行为的展示控件冒充可执行功能。
- **FR-058 常驻上下文**：页面应持续展示当前作业、当前正式版本及来源、当前对话、对话依据版本、课程资料数量和待处理问题数。

## 5. 数据需求

核心实体为 `team`、`team_members`、`assignments`、`courses`、`materials`、`answer_versions`、`candidate_drafts`、`reviews`、`optimization_tasks`、`conversations` 和 `attachments`。系统只有一个小组，但可有多道作业题；当前课程、当前题目与当前对话独立选择。每道题拥有一条正式答案版本链且只有一个当前正式版本；候选稿可并存但不能充当正式答案；对话和优化任务均绑定明确的基础版本。

文件数据至少应保证：标识唯一、引用完整、小组单例约束、题目间答案隔离、每题版本唯一且单调递增、成员操作可追踪、课程资料正文可恢复、时间字段统一记录。

## 6. 非功能需求

- **NFR-001 正确性**：100% 的当前课程检索调用包含 `active_course_id` 过滤；历史检索调用必须记录触发原因。
- **NFR-002 可追踪性**：面向用户的课程事实或评审依据应尽可能附来源；检索证据可追溯到文件及页码/章节。
- **NFR-003 可靠性**：业务数据写入使用事务；资料校验或本地保存失败保留诊断状态，相同课程和文件哈希不得产生重复记录，重新上传可修复丢失文件。
- **NFR-004 可维护性**：UI、Agent、工具、检索、持久化、资料处理按模块分离，Pydantic 模型集中定义。
- **NFR-005 可测试性**：核心策略可在不启动 Streamlit、不发起真实模型请求的情况下通过替身进行单元测试。
- **NFR-006 安全性**：API Key 仅从环境或安全配置读取，不写入仓库、日志或数据文件；上传文件需校验类型和大小。
- **NFR-007 性能基线**：总正文不超过字符预算时返回范围内全部资料；超过预算时默认最多返回 5 个相关章节。请求不得无条件检索历史资料；生成和优化允许按 FR-053 调用独立审查，但自动修正最多一次，调用上界应可测试。
- **NFR-008 可观测性**：每次用户请求具有可关联的 trace/session 标识，能够定位路由、检索范围和失败步骤。
- **NFR-009 兼容性**：首版目标环境为 Windows 11、Python 3.12。
- **NFR-010 可理解性**：首次使用者不阅读外部说明，也应能从页面文案识别课程资料与作业稿件的区别，并通过首页主操作完成“上传已有稿”或“生成初版”流程。

## 7. 验收标准

1. 上传至少 1 个 `.md` 和 1 个 `.txt` 后，能够按当前课程检索并展示文件名与章节。
2. 准备当前与历史课程中含冲突表述的测试资料，普通作业请求仅使用当前课程；明确要求回顾历史时才返回历史证据。
3. 总结、完成、评审、修改四类请求均路由到正确专业 Agent，并通过 Pydantic 输出校验。
4. 评审结果至少包含总分、维度评分、优点、严重问题、教师可能追问和修改优先级；所有扣分项具备五项解释字段。
5. 同一小组可创建并切换至少两道题；切换后只展示所选题目的当前正式答案，两个题目的正式版本和候选稿互不影响。
6. 同一道题始终只有一个当前正式答案；生成或优化只产生候选稿，用户采用后才能创建关联基础版本、审查和操作成员的新正式版本。
7. 关闭并重新打开应用后，对话、小组、题目列表、当前题目、正式版本和候选稿均可恢复，且对话与业务数据存储职责清晰。
8. 任取一次完整任务，可从 trace 中看到主 Agent 路由、检索工具及过滤范围、专业 Agent、保存步骤和异常信息。
9. 用户可上传一份 `.docx` 作业作为初版，再上传线下修改稿形成下一正式版本；两个版本原件、正文、来源和说明均可恢复。
10. 同一道题可建立两条对话，绑定各自明确的正式基础版本，并分别产生 Agent 方案与用户方案的候选稿；任一对话的消息和候选稿不会泄漏到另一条对话。
11. 生成或优化任务完成前自动执行独立审查；结果先显示为候选稿，只有用户点击采用后才创建下一正式版本。
12. 用户可输入或上传优化方向，也可在没有方向时先获得 Agent 问题分析，再选择建议并创建优化任务。
13. 首次进入和关键空状态均展示用途、当前状态和主操作；从首页最多三次主要操作即可开始上传已有稿或生成初版。

## 8. 约束与假设

- Agents SDK 和 OpenAI-compatible 模型接口的具体 API 以实现时安装版本为准；课程资料检索不依赖外部向量库。
- 第一版采用单机单进程纯文件存储和单小组假设；成员身份用于成果归属和操作记录，不实现账号登录与并发权限控制。
- 用户负责把原始课程资料解析为 UTF-8 文本并保留所需页码/章节；课程资料仍只支持 `.md`/`.txt`。作业稿件和优化方向另行支持 `.docx` 文本提取，但不保证复杂排版、公式和图表语义完整恢复。
