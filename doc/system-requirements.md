# CoursePilot 系统需求规格说明书

## 1. 文档目的

本文将 `doc/task.md` 转化为可验收的系统需求，作为架构设计、模块设计、开发和测试的共同基线。第一版目标是构建一个课程学习 Agent 系统，重点验证意图识别、受控检索、专业 Agent 编排、上下文管理、结构化输出和执行链路观测。

## 2. 产品范围

### 2.1 范围内

- 使用自然语言完成课程笔记总结、作业完成、作业评审和作业修改。
- 由 1 个课程学习主 Agent 统一编排 4 个专业 Agent。
- 默认只检索当前课程资料，满足条件时才检索历史课程资料。
- 上传预先整理好的 Markdown 或 UTF-8 纯文本课程资料，并保留其中的章节和来源信息。
- 使用 JSONL 会话记忆维持多轮上下文，使用 Markdown/YAML 持久化业务记录。
- 对 Agent、模型和工具调用进行链路追踪。

### 2.2 范围外

- 复杂角色权限、组织级租户隔离和生产级计费；首版只支持一个项目小组及其成员。
- LangGraph、人工审批断点和长流程恢复。
- 自动接入教务系统、在线提交作业或替代教师给出正式成绩。
- OCR、音视频转写和复杂公式/图表语义识别。

## 3. 用户与核心场景

主要用户为共同完成一份课程大作业的学生小组。系统围绕一个小组、一份大作业和一份持续迭代的共享答案工作；各阶段课程资料为同一份大作业提供知识依据。

| 场景         | 用户目标                   | 预期结果                               |
| ------------ | -------------------------- | -------------------------------------- |
| 课程资料导入 | 建立可检索课程知识库       | 文件正文、课程元数据、页码和来源保存在本地 |
| 小组初始化   | 建立大作业协作上下文       | 记录小组名称、成员和唯一大作业         |
| 笔记总结     | 提炼当前课程内容并用于实践 | 结构化笔记、课程依据、必要的前序知识   |
| 作业完成     | 结合课程进展完善小组大作业 | 任务理解、共享答案、课程依据、不确定项 |
| 作业评审     | 独立评价小组当前答案版本   | 分项评分、扣分依据、追问和修改优先级   |
| 作业修改     | 按评审迭代小组当前答案版本 | 新版本、逐项修改说明、问题闭环状态     |
| 连续对话     | 继续优化“刚才的答案”     | 正确继承课程、答案版本和最近评审       |

## 4. 功能需求

### 4.1 课程与资料管理

- **FR-001 课程管理**：系统应支持创建、查看和切换课程，并且任一会话至多有一个当前课程。
- **FR-002 元数据管理**：每份资料应至少包含 `course_id`、`course_name`、`course_date`、`teacher`、`topic`、`material_type`、`status`；`status` 取值至少支持 `current` 和 `archived`。
- **FR-003 文件导入**：系统应支持 UTF-8 `.md` 和 `.txt`；课程资料由用户预先解析，系统不处理 PDF/PPTX。
- **FR-004 页码保留**：预处理结果应转换为带页码标题的 Markdown，使检索结果能够定位原文件及页码。
- **FR-005 本地资料持久化**：系统应为上传内容自动添加 YAML Front Matter，并统一保存为独立 Markdown 文件。当前课程资料总量未超过字符预算时直接提供正文；超过预算时按章节执行本地关键词检索，Front Matter 不发送给模型。
- **FR-006 当前课程标记**：课程切换后，检索过滤条件和会话上下文应同步更新，不得继续默认使用旧课程。

### 4.2 小组与唯一大作业

- **FR-007 小组管理**：系统应维护一个项目小组的名称和成员名单；首版不支持多个小组并行管理。
- **FR-008 唯一大作业**：系统全局只维护一份小组大作业，不提供创建多个作业的能力；初始化后只能读取或更新该大作业的题目、要求和评分标准。
- **FR-009 共享成果**：答案、评审和修改均属于该小组大作业的共享成果；每次修改产生共享答案的新版本，而不是创建新作业。

### 4.3 受控检索

- **FR-010 当前课程检索**：提供 `search_current_course(query)`，强制以会话中的 `active_course_id` 过滤资料。
- **FR-011 历史课程检索**：提供 `search_course_archive(query)`，排除当前课程并限定归档资料。
- **FR-012 当前优先策略**：所有学习任务默认先检索当前课程，不允许专业 Agent 直接无过滤检索全部资料。
- **FR-013 历史检索条件**：只有用户明确要求、当前资料引用前序知识、当前资料不足或修改需要检查跨课程一致性时，才允许历史检索。
- **FR-014 当前标准优先**：历史资料不得覆盖当前课程教师给出的定义、标准和任务要求。
- **FR-015 检索证据**：检索结果应携带文件名、课程、页码或章节、片段等可引用信息。

### 4.4 Agent 编排

- **FR-020 意图识别**：主 Agent 应识别总结、完成、评审、修改等意图；缺少关键输入时应指出缺失项。
- **FR-021 课程确定**：主 Agent 应从显式输入或会话上下文确定当前课程；无法可靠确定时不得猜测。
- **FR-022 专业 Agent 调用**：主 Agent 应将 4 个专业 Agent 作为工具调用，并保留最终控制权和响应整合职责。
- **FR-023 多 Agent 组合**：主 Agent 可按任务需要顺序调用多个专业 Agent，例如先评审再修改。
- **FR-024 笔记总结**：笔记 Agent 应生成课程问题、核心概念、分析方法、案例、常见错误、教师标准、实践用法和前序关系，并标注来源。
- **FR-025 作业完成**：作业 Agent 应读取唯一大作业及小组当前答案版本，先分析任务和课程标准，再补充或生成共享答案并对题目自检；输出任务理解、共享答案、课程依据和不确定信息。
- **FR-026 独立评审**：评审 Agent 仅接收题目、当前课程资料、教师要求、评分标准和待评答案，不接收生成 Agent 的中间推理。
- **FR-027 评审可解释性**：每项扣分应给出分值、位置、依据、原因和修改建议。
- **FR-028 作业修改前置条件**：修改 Agent 必须读取小组当前答案版本和对应评审结果；不存在评审结果时应先触发评审或提示用户。
- **FR-029 修改模式**：修改应支持“保守修改”和“深度重构”，输出修改稿和逐项修改说明，并复查关键问题是否解决。

### 4.5 上下文与业务记录

- **FR-030 会话上下文**：系统应维护 `active_course_id`、`active_course_name`、`team_id`、`assignment_id`、`current_answer`、`latest_review` 和 `answer_version`；大作业标题和要求从唯一业务记录读取。
- **FR-031 会话持久化**：多轮对话使用自定义 JSONL Session，重启后能够恢复并支持追加、弹出和清空消息。
- **FR-032 业务持久化**：课程资料、作业和答案正文保存为 Markdown；工作区、课程、评审和修改保存为 YAML；不使用数据库，也不以聊天历史代替业务记录。
- **FR-033 答案版本**：唯一大作业可拥有多个共享答案版本；每次保存修改稿应创建新版本，保留原答案、来源评审、修改模式和操作成员的关联，但不得创建新的作业记录。
- **FR-034 状态一致性**：答案、最近评审和版本号更新应保持一致；失败时不得留下半更新状态。

### 4.6 界面与可观测性

- **FR-040 Streamlit 界面**：应提供小组与成员信息、唯一大作业、课程切换、资料导入、对话、共享答案/评审查看和执行状态反馈；不提供作业列表或新增作业入口。
- **FR-041 结构化输出**：关键 Agent 输出使用 Pydantic 模型校验；校验失败应被捕获并向用户提供可恢复反馈。
- **FR-042 Tracing**：应记录模型生成、专业 Agent 调用、函数工具调用、过滤条件、异常和耗时。
- **FR-043 错误处理**：文件校验、本地保存、检索和模型调用失败时，应展示可理解错误并记录诊断信息。

## 5. 数据需求

核心实体为 `teams`、`team_members`、`assignment`、`courses`、`materials`、`answers`、`reviews`、`revisions`。系统只有一个小组和一份大作业；课程只拥有资料，不拥有作业。唯一大作业拥有多个共享答案版本；一个答案版本可有多个评审；一次修改引用原答案版本和评审并产生新答案版本。

文件数据至少应保证：标识唯一、引用完整、小组和大作业的单例约束、答案版本唯一、成员操作可追踪、课程资料正文可恢复、时间字段统一记录。

## 6. 非功能需求

- **NFR-001 正确性**：100% 的当前课程检索调用包含 `active_course_id` 过滤；历史检索调用必须记录触发原因。
- **NFR-002 可追踪性**：面向用户的课程事实或评审依据应尽可能附来源；检索证据可追溯到文件及页码/章节。
- **NFR-003 可靠性**：业务数据写入使用事务；资料校验或本地保存失败保留诊断状态，相同课程和文件哈希不得产生重复记录，重新上传可修复丢失文件。
- **NFR-004 可维护性**：UI、Agent、工具、检索、持久化、资料处理按模块分离，Pydantic 模型集中定义。
- **NFR-005 可测试性**：核心策略可在不启动 Streamlit、不发起真实模型请求的情况下通过替身进行单元测试。
- **NFR-006 安全性**：API Key 仅从环境或安全配置读取，不写入仓库、日志或数据文件；上传文件需校验类型和大小。
- **NFR-007 性能基线**：总正文不超过字符预算时返回范围内全部资料；超过预算时默认最多返回 5 个相关章节。单次普通请求避免无条件调用历史检索和多个专业 Agent。
- **NFR-008 可观测性**：每次用户请求具有可关联的 trace/session 标识，能够定位路由、检索范围和失败步骤。
- **NFR-009 兼容性**：首版目标环境为 Windows 11、Python 3.12。

## 7. 验收标准

1. 上传至少 1 个 `.md` 和 1 个 `.txt` 后，能够按当前课程检索并展示文件名与章节。
2. 准备当前与历史课程中含冲突表述的测试资料，普通作业请求仅使用当前课程；明确要求回顾历史时才返回历史证据。
3. 总结、完成、评审、修改四类请求均路由到正确专业 Agent，并通过 Pydantic 输出校验。
4. 评审结果至少包含总分、维度评分、优点、严重问题、教师可能追问和修改优先级；所有扣分项具备五项解释字段。
5. 系统只能初始化一份小组大作业；再次初始化会被拒绝或转为更新，界面没有新增第二份作业的入口。
6. 修改请求没有评审时不会直接重写；存在评审时能生成共享答案新版本并关联原版本和操作成员。
7. 关闭并重新打开应用后，会话、小组、唯一大作业和共享成果均可恢复，且会话与业务数据存储职责清晰。
8. 任取一次完整任务，可从 trace 中看到主 Agent 路由、检索工具及过滤范围、专业 Agent、保存步骤和异常信息。

## 8. 约束与假设

- Agents SDK 和 OpenAI-compatible 模型接口的具体 API 以实现时安装版本为准；课程资料检索不依赖外部向量库。
- 第一版采用单机单进程纯文件存储和单小组假设；成员身份用于成果归属和操作记录，不实现账号登录与并发权限控制。
- 用户负责把原始资料解析为 UTF-8 文本并保留所需页码/章节；系统只负责校验、存储和检索。
